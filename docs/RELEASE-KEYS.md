# Release Keys

Authoritative reference for the cryptographic identities Caloura ships with.
Treat this file as the **only** approved entry point for rotating, recovering, or
verifying the keys; nothing else in the repo should imply different values.

## 1. Sparkle EdDSA signing key (auto-update)

Sparkle gates every auto-update by verifying that the `sparkle:edSignature`
attribute on the appcast `<enclosure>` was produced by the private key whose
public counterpart is embedded in the shipping app. If the public key in the
app does not match the key that signed the update, Sparkle refuses the
download — the feed is effectively bricked for that user until they manually
install a build with the correct embedded public key.

### Embedded public key (current production)

| Location | Key |
|---|---|
| `Caloura/Resources/Info.plist` (`SUPublicEDKey`) | `jvH62SR8OsHYcY9lPGDneNLbOWijs5GGempwmXnEUL8=` |
| `project.yml` (`info.properties.SUPublicEDKey`) | `jvH62SR8OsHYcY9lPGDneNLbOWijs5GGempwmXnEUL8=` |

The `Info.plist` value is what Sparkle reads at runtime. The `project.yml`
value is the source of truth that XcodeGen regenerates the plist from. Both
must match. CI ships from the regenerated plist; do not hand-edit the plist.

### Origin

The keypair was generated by Sparkle's `generate_keys` tool on the original
release engineer's machine in February 2026 (see archived plan
`tasks/archive/plan-legacy-2026-02-04.md`, commit `e8bc0fd`). The private key
is stored in that machine's macOS Keychain under the item Sparkle creates
automatically. The `sign_update` binary that signs each release reads the
private key from Keychain — there is no exported file.

`sign_update` is invoked from `scripts/publish.sh` which resolves it from
either `.build/xcode/SourcePackages/artifacts/sparkle/Sparkle/bin/sign_update`
(when the project has been built) or `~/Applications/Sparkle/bin/sign_update`
(installed copy).

### Rotation procedure (DO NOT do this casually)

Rotating the EdDSA key has a one-shot escape hatch: the **first** build that
ships a new public key must be installed manually by every existing user,
because their currently-installed app still trusts the old key. Sparkle has
no built-in key-rollover protocol.

If rotation is genuinely required (compromise, lost Keychain, etc.):

1. Run `./bin/generate_keys` from the Sparkle distribution. This stores the new
   private key in Keychain and prints the new public key.
2. Update `project.yml` `SUPublicEDKey` and regenerate the project
   (`xcodegen`).
3. Verify `Caloura/Resources/Info.plist` `SUPublicEDKey` now matches.
4. Bump the marketing version + build number.
5. Build, sign, notarize, and **publish via the website only** — do not push
   the appcast yet. Existing users will not see this update.
6. Update `caloura.app` landing page with a banner directing users to download
   the new build manually.
7. Once enough of the user base has installed the new build manually, resume
   publishing via `./scripts/publish.sh`. New signatures will validate against
   the new public key.
8. Update this file's "Embedded public key" table with the new value.

### Recovery if the private key is lost

If the macOS Keychain on the release machine is wiped without exporting the
private key, the keypair is **unrecoverable**. Follow the rotation procedure
above. Until that completes, no auto-update can be shipped — only manual
download.

To prevent this scenario, the release machine's Keychain should be backed up
to an encrypted external drive after every key generation. The backup file
must never be committed to a repo, sent over email, or stored on a cloud
service.

## 2. License entitlement public key

Caloura's per-user license entitlements are signed by a server-side private
key. The shipping app verifies entitlements with a public key supplied at
build time via the `CALOURA_LICENSE_ENTITLEMENT_PUBLIC_KEY` environment
variable, which `project.yml` writes into the Info.plist as
`CalouraLicenseEntitlementPublicKey`.

| Location | Source |
|---|---|
| `Info.plist.CalouraLicenseEntitlementPublicKey` | env `CALOURA_LICENSE_ENTITLEMENT_PUBLIC_KEY` at build time |
| `Info.plist.CalouraLicenseEntitlementURL` | env `CALOURA_LICENSE_ENTITLEMENT_URL` at build time |

The corresponding private key lives only on the entitlement-issuing server
and is **never** stored in this repo, in any developer's machine, or in CI.
Rotation is server-side; the client only changes when the embedded public key
needs to be updated for the next release.

## 3. Apple developer / notarization identities

These are not embedded in the shipping app but are required to produce
notarized builds.

| Identity | Where it lives |
|---|---|
| Developer ID Application certificate | macOS Keychain on release machine, login keychain |
| Notarization keychain profile | `Caloura-Notarize` (login keychain) |

Both are managed via Xcode + `xcrun notarytool`. See `scripts/release.sh` for
the end-to-end notarization flow.

## 4. Verifying the chain after a release

After `./scripts/publish.sh <version>` completes, verify the live appcast
matches the published artifact:

```
python3 scripts/validate_appcast_against_manifest.py \
    --manifest build/release-manifest-<version>.json \
    --url https://caloura.app/appcast.xml
```

This re-runs `sign_update` against the local Sparkle artifact and confirms
the signature embedded in the live appcast matches. The publish script runs
this automatically with retries; the manual command is for ad-hoc audits.
